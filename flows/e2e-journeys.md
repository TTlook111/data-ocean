# 端到端用户旅程

跨模块的完整用户流程，确保模块间衔接顺畅。

---

## 旅程 1：管理员首次配置系统

**角色**: SUPER_ADMIN（系统初始化后的唯一账号）

```
登录页 → 强制改密 → 管理首页
  → 创建部门结构（部门树页）
  → 创建角色（预置，无需操作）
  → 创建用户并分配角色/部门（用户管理页）
  → 添加数据源并测试连接（数据源管理页）
  → 给用户授权数据源（数据源授权弹窗）
  → 通知用户可以登录使用
```

**关键衔接点**:
- 创建用户时，部门下拉必须能看到刚创建的部门（无需刷新页面）
- 数据源授权弹窗的用户列表必须包含刚创建的用户
- 整个流程不应需要刷新浏览器

---

## 旅程 2：新用户首次登录到发起查询

**角色**: 普通用户（DATA_ANALYST / BUSINESS_USER）

```
收到账号 → 登录页输入初始密码 → 强制改密页
  → 改密成功自动退出 → 重新登录
  → 进入问答端数据源选择页
  → 看到被授权的数据源卡片
  → 选择一个数据源 → 进入查询对话页（后续模块）
```

**关键衔接点**:
- 强制改密后必须退出重新登录（安全要求，JWT 需要刷新）
- 普通用户登录后默认进入 `/query`（不是 `/admin`）
- 如果没有任何授权数据源，显示空状态引导联系管理员
- 数据源卡片需要展示足够信息让用户识别（名称 + 库名 + 描述）

---

## 旅程 3：管理员日常运维 — 处理用户锁定

**角色**: SUPER_ADMIN / ADMIN

```
收到用户反馈"登录不了" → 登录管理后台
  → 用户管理页 → 筛选该用户（按用户名或姓名）
  → 看到状态为"已锁定" → 点击"解锁"
  → 确认弹窗 → 解锁成功 → 状态变为"正常"
  → （可选）重置密码 → 通知用户
```

**关键衔接点**:
- 筛选应支持模糊搜索，输入即过滤（防抖 300ms）
- 解锁操作需要二次确认（防误操作）
- 解锁后表格行状态即时更新，无需手动刷新

---

## 旅程 4：管理员停用数据源

**角色**: ADMIN

```
发现某数据源有问题 → 数据源管理页
  → 点击"禁用" → 确认弹窗（提示影响：N个用户将无法查询）
  → 禁用成功 → 状态标签变灰
  → 此时用户端：该数据源卡片消失
  → 问题修复后 → 点击"启用" → 数据源恢复可见
```

**关键衔接点**:
- 禁用确认弹窗应告知影响范围（当前有多少用户被授权）
- 禁用后用户端实时生效（下次请求即不返回该数据源）
- 如果用户正在使用该数据源查询，应有优雅降级（提示数据源已下线）

---

## 旅程 5：用户自助修改个人信息

**角色**: 任意已登录用户

```
点击顶栏用户头像/名称 → 个人资料页
  → 修改姓名/邮箱/手机 → 保存
  → 顶栏显示的用户名实时更新
  → 点击"修改密码"链接 → 跳转改密页
  → 输入旧密码 + 新密码 → 修改成功 → 自动退出重新登录
```

**关键衔接点**:
- 保存个人资料后，AppShell 顶栏的用户名应同步更新（不刷新页面）
- 修改密码成功后必须退出（JWT 失效策略）
- 改密页需要区分"强制改密"和"自愿改密"两种模式的 UI 提示
