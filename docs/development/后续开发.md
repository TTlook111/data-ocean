### 当前建议执行顺序

截至 2026-06-21，主链路已经闭环，后续修改优先按“补齐可见管理能力 → 补齐后台真实执行 → 增强工程化能力 → 远期扩展”的顺序推进：

1. **第一批：低耦合补齐项** — P0 管理员反馈特权。
2. **第二批：后台能力补全** — P9 告警触发执行逻辑、P10 配额维度扩展。
3. **第三批：管理端增强** — P12 告警规则和配额前端管理。建议等 P9/P10 后实施，避免出现“前端能配置但后端不真正执行”的体验。
4. **第四批：工程化增强** — P5 Redis 记忆、P6 可观测性、Agent workflow 测试补强。
5. **暂缓项** — P8 多实例部署、P3 PostgreSQL、P4 图数据库血缘，除非进入上线部署或性能瓶颈阶段。

### P5 — Agent 短期记忆 Redis 化

**背景**：当前 Python Agent 的对话历史完全依赖 Java 端每次从 MySQL 查询后传入（最多 5 轮），`user_memory` 硬编码为 None。Python 服务虽声明了 `redis_host`/`redis_port` 配置但无任何 Redis 客户端代码。需要引入 Redis 实现对话历史缓存、会话上下文、用户偏好和最近查询的短期记忆。

**设计边界**：Java 已经接入 Redis，且 MySQL 会话消息由 Java 负责持久化。Redis 记忆不要让 Java 和 Python 同时写同一个 conversation history key，否则容易出现顺序不一致和重复写入。建议分工如下：

- Java 负责 `agent:conv:{conversationId}:history` 的读穿透缓存，因为消息的真实来源是 `ConversationService`/MySQL。
- Python 负责派生记忆（会话意图、最近表、用户偏好、最近查询摘要），用于增强 Query Rewrite 和后续追问。
- Python 读取 `user_memory` 时优先读派生记忆；对话原文仍以 Java 请求里传入的 `conversation_history` 为准。

**Redis Key 设计**：

```
# 对话历史缓存（List，原子 LPUSH + LTRIM）
agent:conv:{conversationId}:history
    type: List
    element: {"role":"user","content":"..."} (JSON string per element)
    max_length: 20 elements (10轮)
    TTL: 2 hours

# 会话级上下文摘要（Hash，HSET 单字段更新）
agent:conv:{conversationId}:context
    type: Hash
    fields: last_tables(JSON), last_intent(JSON), last_sql, query_count, updated_at
    TTL: 2 hours

# 用户级偏好记忆（Hash，HSET 单字段更新）
agent:user:{userId}:prefs
    type: Hash
    fields: preferred_datasource, preferred_chart, frequent_tables(JSON), frequent_metrics(JSON), last_active
    TTL: 7 days

# 用户最近查询摘要（List，原子 LPUSH + LTRIM）
agent:user:{userId}:recent_queries
    type: List
    element: {"question":"...","tables":["..."],"intent":"...","ts":...}
    max_length: 20
    TTL: 7 days
```

**并发安全设计**：
- 对话历史使用 `List` + `pipeline(LPUSH + LTRIM + EXPIRE)` 原子操作，避免读-改-写竞态
- 用户偏好/会话上下文使用 `HSET` 单字段更新，不整体覆盖
- Redis 连接池初始化使用 `asyncio.Lock` + double-check 防并发初始化
- 所有操作 try-catch 静默降级，Redis 故障不阻断查询

**改动范围**：

1. Python 侧：
   - 新增 `python-service/dataocean/infra/memory.py`：Redis 连接池 + 4 类记忆的 CRUD 操作
   - 修改 `python-service/dataocean/core/config.py`：补充 `redis_password`、`redis_db` 配置项
   - 修改 `python-service/pyproject.toml`：新增 `redis>=5.0.0` 依赖
   - 修改 `python-service/dataocean/agent/nodes/query_rewriter.py`：`_build_variables` 中从 Redis 读取派生的 `user_memory` 替代硬编码 None
   - 修改 `python-service/dataocean/agent/router.py`：Agent 执行成功后异步写入会话上下文、用户偏好、最近查询摘要

2. Java 侧：
   - 修改 `PythonAgentClientImpl.buildConversationHistory()`：先查 Redis 缓存，未命中再查 MySQL，结果写回 Redis
   - 修改 `ConversationServiceImpl.deleteConversation()`：同时清除 Redis 缓存
   - 检查 Java/Python Redis DB、key 前缀、TTL 配置是否一致，避免不同服务写到不同逻辑库

3. Prompt 模板：
   - 修改 `query_rewrite.j2`：已有 `user_memory` 占位，确保 Python 侧传入后正确渲染

**涉及文件**：

### P6 — 可观测性建设

**背景**：当前系统无 Metrics/监控、无链路追踪、无结构化日志。日志覆盖率 71% 但为非结构化中文硬编码消息，无 request_id/task_id 全局传播。LangChain tracing 配置预留但默认关闭。生产环境出问题后无法通过监控发现、无法通过 trace 定位。

**改动范围**：

1. 结构化日志：
   - 修改 `python-service/dataocean/core/logging.py`：改用 JSON 格式输出，注入 task_id/user_id 全局上下文
   - 引入 `structlog` 或 Python 标准 `logging` 的 JSON Formatter
   - 在 `agent/router.py` 的请求入口设置 task_id 上下文变量

2. Metrics/监控：
   - 新增 `python-service/dataocean/infra/metrics.py`：Prometheus 指标定义
   - 核心指标：查询总数/成功数/失败数（counter）、查询耗时（histogram）、LLM 调用次数/耗时（counter/histogram）、RAG 召回命中率（gauge）、SQL 执行耗时（histogram）、活跃任务数（gauge）
   - 新增 `GET /metrics` 端点（或集成 `prometheus-fastapi-instrumentator`）
   - Java 侧可选：集成 Micrometer + Prometheus

3. 链路追踪：
   - 启用 LangChain tracing（`LANGCHAIN_TRACING_V2=true`），配置 LangSmith 或接入 OpenTelemetry
   - 在 Java→Python 的 HTTP 请求中传递 trace_id header
   - Python 侧在 AgentState 中携带 trace_id，贯穿所有节点

4. 健康检查增强：
   - 修改 `python-service/dataocean/infra/health.py`：区分 readiness/liveness 端点
   - readiness 检查 Milvus + LLM + 连接池状态
   - liveness 仅检查进程存活
   - 返回版本号、启动时间等元数据

**涉及文件**：
- 修改 `python-service/dataocean/core/logging.py`
- 新增 `python-service/dataocean/infra/metrics.py`
- 修改 `python-service/dataocean/main.py`（注册 metrics 端点）
- 修改 `python-service/dataocean/infra/health.py`

### P8 — 多实例部署支持

**背景**：当前 Python 服务有 3 处进程内状态为水平扩展的致命阻塞：SSE 事件队列（`infra/sse.py` 的 `_event_queues`）、取消令牌（`infra/cancellation.py` 的 `_cancelled_tasks`）、活跃任务引用（`agent/router.py` 的 `_active_tasks`）。Dockerfile 默认单 worker，无 K8s 配置。

**改动范围**：

1. 短期方案 — 会话亲和：
   - 在负载均衡层（Nginx/K8s Ingress）配置基于 task_id 的会话亲和
   - Java 端在请求头中传递 task_id
   - 连接池上限按实例数均分（`pool_max_global / N`）

2. 长期方案 — 外部化状态：
   - SSE 事件队列 → Redis Pub/Sub 或 Redis Streams
   - 取消令牌 → Redis SET with TTL
   - 活跃任务引用 → 改为"标记取消 + 轮询检测"模式（asyncio.Task 无法外部化）
   - 连接池 → 保持进程内，通过 Redis 做全局连接数协调
   - 配置热重载 → 逐实例回调或共享配置源

3. 部署配置：
   - 新增 `python-service/Dockerfile`（多阶段构建）
   - 修改 `docker-compose.yml`：添加 python-service 和 java-backend 服务定义
   - 可选：新增 K8s manifests 或 Helm charts

**涉及文件**：
- 修改 `python-service/dataocean/infra/sse.py`
- 修改 `python-service/dataocean/infra/cancellation.py`
- 修改 `python-service/dataocean/agent/router.py`
- 修改 `python-service/dataocean/sandbox/pool_manager.py`

### P9 — 告警触发执行逻辑

> 近期功能增强任务，不是占位项；P12 的告警历史页面需要依赖这里的执行与历史数据。

**背景**：后端 `alert_rule` 表和 `AlertRuleController` 已完整实现（CRUD + toggle），但缺少定时扫描 metric 并触发通知的实际执行逻辑。告警规则定义了 ERROR_RATE / SLOW_QUERY_COUNT / COST_DAILY 三种监控指标和阈值，但无后台任务去比对和触发。

**改动范围**：

1. 新增 `AlertEvaluateJob` 定时任务（每 5 分钟执行）：
   - 读取所有启用的 `AlertRule`
   - 按 `metric_type` 查询对应指标（ERROR_RATE 查 query_audit_log 的成功率、SLOW_QUERY_COUNT 查慢查询数、COST_DAILY 查 llm_usage_log 的当日费用）
   - 与阈值比较，触发时调用 `NotificationService.send()` 发通知
   - 记录告警历史（避免重复告警：同一规则在恢复前只告警一次）

2. 新增 `alert_history` 表：
   - Flyway 迁移脚本
   - 字段：rule_id、triggered_at、resolved_at、metric_value、threshold_value

**涉及文件**：
- 新增 `backend/.../audit/service/impl/AlertEvaluateJob.java`
- 新增 `backend/.../audit/entity/AlertHistory.java`
- 新增 `backend/.../audit/mapper/AlertHistoryMapper.java`
- 新增 `src/main/resources/db/migration/V43__create_alert_history.sql`（当前迁移已到 V42，实际开发时继续使用下一个未占用版本号）
