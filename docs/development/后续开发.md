## 开发流程

1. 代码实现并提交
2. **更新 CLAUDE.md** — 如果改了架构、模块职责、技术栈、API 边界

一致性检查方法：让 AI Agent 读 CLAUDE.md 的每一项声明，与代码交叉验证（技术栈版本、模块列表、API 端点、枚举值、测试数量等）。

## 已确认的开发任务

### P0 — 管理员反馈特权（反滥用机制补全）

**背景**：设计文档要求管理员/分析师的"踩"直接生效且权重为普通用户 3 倍，当前实现中所有用户走相同流程，未区分角色。

**改动范围**：

- `UserFeedbackServiceImpl.submitFeedback()` 增加角色判断逻辑
- 管理员/分析师的 DISLIKE 跳过审核队列，直接触发 `USER_DISLIKE_CONFIRMED` 事件，delta 使用 -45（普通用户 -15 的 3 倍）
- 普通用户流程不变（限流 → 审核队列 → 组阈值检测）

**涉及文件**：
- `backend/.../fieldtag/service/impl/UserFeedbackServiceImpl.java`
- `backend/.../fieldtag/service/impl/FeedbackReviewServiceImpl.java`

**文档更新**：完成后更新 CLAUDE.md 的 fieldtag 模块描述（如行为变了）

---

### P1 — 通知系统完善（前端接入 + 后端事件对接）

**背景**：后端通知系统已完整实现（`NotificationController` 提供 3 个 REST 端点），但前端零接入。同时有 2 个事件发布后无人消费。

**改动范围**：

1. 前端接入：
   - 新增 `frontend/src/api/notification.ts`，封装 3 个 API
   - 在 `AppShell.vue` 顶部栏添加通知铃铛图标 + 未读角标 + 下拉通知列表

2. 后端事件对接：
   - 新增 `GroupThresholdAlertEventListener`：监听 `GroupThresholdAlertEvent`（3 用户踩同字段），调用 `NotificationService.send()` 发通知给管理员
   - 扩展 `SnapshotPublishedEventListener`：快照发布/过期时调用 `NotificationService.send()` 发通知给相关用户（当前只有 `log.info()`）

**涉及文件**：
- 新增 `frontend/src/api/notification.ts`
- 修改 `frontend/src/components/AppShell.vue`
- 新增 `backend/.../fieldtag/event/GroupThresholdAlertEventListener.java`
- 修改 `backend/.../versioning/event/SnapshotPublishedEventListener.java`

**文档更新**：完成后更新 AGENTS.md 的前端 API 模块列表和路由树
