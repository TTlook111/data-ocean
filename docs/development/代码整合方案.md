当前项目**尚未满足**本方案的大部分整合目标。已满足的是 RestClient 已集中配置为 3 个 Bean；未满足的是 JDBC URL 统一、Python 客户端公共错误处理、AI 配置分层、权限校验提取、CollectorContext、PythonAgentClient 拆分、常量/工具方法提取等。

原始方案识别的问题方向基本正确。我之前重写时确实把一部分“低优先级/有争议的设计”删掉或弱化了，这不符合“漏洞还存在就要保留”的原则。本版恢复这些问题，但会明确标注：有些问题必须修，有些应作为技术债保留，有些原方案设计方式需要调整。

本次按最新版代码复核后确认：方案中的主要问题仍然存在，但有几项设计表述需要降级或澄清。`vectorize()` 是否重试属于幂等性评估，不应列为 P0 Bug；`notifyPythonReload()` 失败只 warn 是当前 best-effort 语义，不应强行改成重试；`testProvider` 与 `detectDimension` 的错误处理属于 AI 配置模块拆分时顺手收敛的问题，优先级应低于 JDBC 与 Python 客户端公共异常处理。

新增复核还发现 5 类问题：默认异步执行器配置不明确、全局异常未区分连接拒绝和读超时、Python 健康状态对象可变且跨线程共享、Python 连接池客户端重复且状态异常只记录日志、RestClient 缺少连接复用能力。这些问题需要补充到实施计划中。

## 二、设计原则

1. **漏洞不删除**  
   只要代码里仍存在重复、潜在 Bug 或分层问题，文档必须保留。可以调整优先级和方案，但不从清单里消失。

2. **重构不改变业务语义**  
   API 路径、请求/响应结构、权限表达式、Python 内部接口、密钥加解密、脱敏策略、查询取消语义都保持不变。

3. **先修稳定性，再做结构优化**  
   JDBC 超时、Python 调用超时识别、全局超时判断优先于减少行数。

4. **抽象要有边界**  
   优先提取有业务边界的组件，如 `PythonClientSupport`、`PythonAiConfigClient`、`PermissionValidationSupport`。小工具类可以保留为 P2，但不要让工具类泛滥影响可读性。

5. **每阶段可独立验证**  
   每个阶段都应能编译、能测试、能回滚。

## 三、P0 稳定性与高价值整合

### 3.1 默认异步执行器配置不明确

#### 当前状态

项目未满足。当前 `AsyncConfig` 只定义了名为 `queryExecutor` 的线程池：

```java
@Bean("queryExecutor")
public Executor queryExecutor() { ... }
```

`PythonAgentClientImpl` 显式使用了：

```java
@Async("queryExecutor")
```

但项目中还有多个无参 `@Async`：

- `AuditLogServiceImpl`
- `OperationLogServiceImpl`
- `LineageServiceImpl`
- `SnapshotPublishedEventListener`
- `SchemaCollectionServiceImpl`

由于没有实现 `AsyncConfigurer`，也没有名为 `taskExecutor` 的默认执行器，未指定 executor 的 `@Async` 方法不应依赖 Spring 的默认解析行为。最坏情况下会退回到 `SimpleAsyncTaskExecutor`，该执行器不做线程池复用，高并发时可能持续创建新线程。

#### 推荐方案

二选一：

1. 实现 `AsyncConfigurer#getAsyncExecutor()`，将默认异步执行器显式指向受控线程池。
2. 新增一个默认 `taskExecutor`，并让无参 `@Async` 使用它；查询类长耗时任务继续显式使用 `queryExecutor`。

建议拆分两个线程池：

- `queryExecutor`: NL2SQL 查询执行，容量较大，保留当前配置。
- `taskExecutor` 或 `systemAsyncExecutor`: 审计、操作日志、事件监听等轻量后台任务，容量较小。

#### 验收标准

- 所有无参 `@Async` 都能落到受控线程池。
- 不再依赖 `SimpleAsyncTaskExecutor`。
- 查询执行仍使用 `queryExecutor`，避免被审计/日志任务占满。

### 3.2 JDBC URL 构建统一

#### 当前状态

项目未满足。当前仍有三处手写 MySQL JDBC URL：

- `SchemaCollectionServiceImpl`: 有 `connectTimeout=10000`、`socketTimeout=60000`
- `SchemaStatisticsServiceImpl`: 缺少 `connectTimeout` 和 `socketTimeout`
- `DatasourceConnectionServiceImpl`: 有 5s 快速失败和安全参数

`SchemaStatisticsServiceImpl` 缺少超时参数是实际稳定性漏洞，真实数据源网络异常时可能长时间阻塞。

#### 原方案设计问题

原方案建议 `JdbcUtils.buildSecureMySqlUrl()` 使用统一 10s/60s 超时，这不适合连接测试。连接测试和健康检查需要快速失败，应保留 5s 策略。

原方案的静态工具类示例里调用了 `log.warn()`，但没有声明 logger，也没有说明 `charset` 的 `utf8mb4 -> utf8` 映射，开发时需要补齐。

#### 推荐方案

新增 `com.dataocean.common.util.JdbcUrlBuilder`，按场景区分：

```java
public final class JdbcUrlBuilder {

    public static String metadataMysqlUrl(String host, Integer port, String databaseName, String charset) {
        // 元数据采集/统计采集：connectTimeout=10000, socketTimeout=60000
    }

    public static String connectionTestMysqlUrl(String host, Integer port, String databaseName, String charset) {
        // 连接测试/健康检查：connectTimeout=5000, socketTimeout=5000
        // 保留 allowLoadLocalInfile=false、autoDeserialize=false、allowUrlInLocalInfile=false
    }

    public static String resolveJdbcEncoding(String charset) {
        // 空值或 utf8mb4 -> utf8，其余保留
    }
}
```

#### 待修改文件

- `backend/DataOcean/src/main/java/com/dataocean/common/util/JdbcUrlBuilder.java`
- `backend/DataOcean/src/main/java/com/dataocean/module/metadata/service/impl/SchemaCollectionServiceImpl.java`
- `backend/DataOcean/src/main/java/com/dataocean/module/metadata/service/impl/SchemaStatisticsServiceImpl.java`
- `backend/DataOcean/src/main/java/com/dataocean/module/datasource/service/impl/DatasourceConnectionServiceImpl.java`

#### 验收标准

- `SchemaStatisticsServiceImpl` 的 URL 包含 `connectTimeout` 和 `socketTimeout`
- 连接测试仍为 5s 快速失败
- 连接测试 URL 仍包含安全参数
- 字符集处理不退化

### 3.3 `PythonRagClientImpl` 超时处理缺陷

#### 当前状态

项目未满足。`PythonRagClientImpl` 当前缺少 `isReadTimeout()`，导致：

- `vectorize()` 把所有 `ResourceAccessException` 都报成“服务暂时不可用”
- `chunkDocument()` 把读超时也包装成 `PythonRetryableException` 触发重试

这会导致错误提示不准确，也可能让超时任务重复压服务。

#### 补充评估：vectorize 与 chunkDocument 的 @Retryable 不对称性

当前 `chunkDocument()` 有 `@Retryable(retryFor = PythonRetryableException.class, maxAttempts = 2)`，但 `vectorize()` 没有 `@Retryable`。

这不是 P0 Bug，而是需要确认的设计取舍：

1. `chunkDocument()` 的重试逻辑：Python 端切割可能因临时资源不足失败，重试有意义
2. `vectorize()` 的重试逻辑：Python 端向量化可能因 Milvus 临时不可达失败，重试也有意义
3. 但向量化可能涉及批量写入 Milvus，需要先确认接口幂等性、版本覆盖策略和失败回滚策略

**推荐方案**：
- 在 `PythonClientSupport` 统一异常处理后，再评估 `vectorize()` 是否需要 `@Retryable`
- 如果向量化操作是幂等的（重复执行不会产生副作用），建议添加 `@Retryable`
- 如果向量化操作非幂等，保持当前设计，并在代码注释中说明“向量化不自动重试”的原因

#### 推荐方案

不要只在 `PythonRagClientImpl` 私有补一个 `isReadTimeout()`，因为 `PythonKnowledgeClientImpl`、`GlobalExceptionHandler` 也有类似判断。应在 P0-3 的 `PythonClientSupport` 里统一：

```java
public static boolean isReadTimeout(Throwable exception) {
    // 遍历 cause 链识别 SocketTimeoutException
    // 兼容 message 中 read timed out / timeout
}
```

然后：

- `vectorize()` 读超时返回“RAG 向量化超时，请稍后重试”
- `chunkDocument()` 读超时返回 `BusinessException`，非读超时再返回 `PythonRetryableException`
- 保持 `deleteDocVersionVectors()` 尽力清理，不改成强失败

### 3.4 Python 客户端公共错误处理

#### 当前状态

项目未满足。当前仍存在：

- `PythonRagClientImpl.toPythonStatusException()`
- `PythonKnowledgeClientImpl.toPythonStatusException()`
- `PythonKnowledgeClientImpl.isReadTimeout()`
- 重复 `@Retryable(retryFor = PythonRetryableException.class, maxAttempts = 2, backoff = @Backoff(delay = 1000))`
- 相似 catch 链

#### 原方案设计问题

原方案的 `executeWithRetry(Supplier<T> call, String taskDesc, String taskId)` 把上下文固定为 `taskId`，不适合所有场景。知识库生成使用 `snapshotId`，AI 配置使用 providerId，未来其他 Client 也不一定有 taskId。

#### 推荐方案

新增 `com.dataocean.common.client.PythonClientSupport`：

```java
public final class PythonClientSupport {

    public static RuntimeException statusException(HttpStatusCode statusCode, String message) {}

    public static boolean isReadTimeout(Throwable exception) {}

    public static <T> T execute(
            Supplier<T> call,
            Function<ResourceAccessException, RuntimeException> resourceAccessMapper,
            Function<Exception, RuntimeException> genericMapper) {}
}
```

`@PythonServiceRetry` 可保留为 P2。P0 阶段先统一状态码和超时判断，避免为了注解美观扩大改动。

#### 验收标准

- 两个 Client 不再重复定义 `toPythonStatusException()`
- 读超时判断逻辑只有一处
- 5xx 仍触发重试
- 4xx 仍作为业务异常处理
- 各调用点的用户提示保持具体，不统一成含糊文案

### 3.5 `GlobalExceptionHandler` 超时判定

#### 当前状态

项目未满足。当前 `handleResourceAccessException()` 只检查 message：

```java
message.contains("Read timed out") || message.contains("timeout")
```

问题：

- 没有遍历 cause 链识别 `SocketTimeoutException`
- `"timeout"` 字符串匹配过宽，可能误判连接超时、读超时或其他错误
- 没有区分 `ConnectException`（连接拒绝/服务不可达）与 `SocketTimeoutException`（读超时）
- 与 Python Client 内部超时判断不一致

#### 推荐方案

复用 `PythonClientSupport` 中的异常分类能力：

```java
public static boolean isReadTimeout(Throwable exception) {}
public static boolean isConnectFailure(Throwable exception) {}
```

建议语义：

- 读超时 `SocketTimeoutException`: HTTP 504，提示“AI 服务响应超时，请稍后再试”
- 连接拒绝 `ConnectException`: HTTP 503，提示“AI 服务暂时不可用，请稍后再试”
- 其他 `ResourceAccessException`: HTTP 503 或保持当前 504 均可，但需要统一项目错误语义

### 3.6 `AiConfigController` 拆分

#### 当前状态

项目未满足。`AiConfigController` 当前仍直接注入：

- `SysConfigService`
- `DatasourceSecretService`
- `ObjectMapper`
- `pythonShortTimeoutRestClient`
- `pythonRestClient`

并在 Controller 中处理 Provider CRUD、默认值迁移、密钥处理、JSON、Python 调用。

#### 原方案设计问题

1. 原方案建议 `notifyPythonReload()` 私有方法加 `@Retryable`，这在 Spring AOP/Retry 中通常不会生效，因为私有方法且自调用不走代理。
2. 原方案 `catch (ResourceAccessException | SocketException)` 不能可靠捕获 RestClient cause 链里的 socket 异常，应统一用 `PythonClientSupport.isReadTimeout()` 或专门的异常映射。
3. “Python reload 带重试”不应作为本轮必修项。当前语义是 Java 配置保存成功后尽力通知 Python，Python 未启动时只 warn。如果改成强重试或抛错，可能改变用户保存配置的体验。

#### 补充问题：testProvider catch(Exception) 过宽

`testProvider()` 方法使用 `catch (Exception e)` 捕获所有异常，包括：
- `BusinessException`（业务异常，应保留原始错误码）
- `ResourceAccessException`（网络连接问题）
- `NullPointerException`、`ClassCastException` 等编程错误

这会导致：
1. 编程错误被掩盖，难以发现问题
2. `BusinessException` 的原始错误码丢失
3. 所有异常都被统一包装成”供应商连接测试失败”

**推荐修复**：
```java
} catch (BusinessException e) {
    // 业务异常直接上抛，保留原始错误码
    throw e;
} catch (ResourceAccessException e) {
    // 网络连接问题，归类为连接测试失败
    provider.setStatus("failed");
    provider.setLastTestedAt(LocalDateTime.now().toString());
    saveProvider(provider);
    log.warn("供应商连接测试失败 id={} reason={}", id, e.getMessage());
    throw new BusinessException("供应商连接测试失败，请检查配置后重试");
} catch (Exception e) {
    // 未知异常，记录详细日志并转为 500
    log.error("供应商测试发生未预期异常 id={}", id, e);
    throw new BusinessException("供应商测试内部错误");
}
```

#### 补充问题：detectDimension 缺少错误处理

`detectDimension()` 方法直接调用 Python REST 接口，没有本地 try-catch。如果 Python 不可达或返回错误，异常会直接抛到 `GlobalExceptionHandler`，返回 504 或 500，但缺少 AI 配置场景下的友好错误提示。

**推荐修复**：在 `PythonAiConfigClient` 中实现时，增加适当的异常处理和错误提示。

#### 推荐方案

拆分为：

```text
AiConfigController
  - 保留路由、权限注解、Result 包装

AiConfigService
  - 配置读取/更新
  - Provider CRUD
  - 默认值补齐和旧配置键兼容
  - API Key 加密/解密/脱敏
  - Embedding pending/reindex 状态维护

PythonAiConfigClient
  - testProvider()
  - detectDimension()
  - reVectorize()
  - notifyReloadBestEffort()
```

`notifyReloadBestEffort()` 建议仍保持“失败只记录 warn”。如果需要补偿机制，应另开“配置重载失败重试任务/后台状态”需求，不在本轮重构中隐式改变。

#### 验收标准

- `AiConfigController` 不再直接依赖 `RestClient`
- 所有 `/api/admin/system/ai-config/**` 路由不变
- 权限注解不变
- API Key 仍加密存储，响应仍脱敏
- 更新 AI 配置后仍尝试通知 Python reload，失败不影响 Java 保存

### 4.1 `PythonAgentClientImpl` 职责过重

#### 当前状态

- SSE progress/result/error 解析
- 任务状态回写
- 助手消息保存
- Python cancel 调用

另外仍存在：

- SSE 读取循环缺少显式总时长保护，只依赖 RestClient/read timeout 和 Python 心跳行为
- `loadFallbackChunks()` 使用 `inSql()` 拼接 `datasourceId`

#### 原方案设计问题

1. `inSql()` 的 datasourceId 来自 Long 参数，实际注入风险很低，但字符串拼 SQL 仍不是好习惯。原方案建议直接用 `LambdaQueryWrapper.in(subqueryWrapper)` 不成立，MyBatis-Plus 的 `in()` 不能直接传子查询 wrapper。
2. 修复方式应为“先查已发布 docId，再 `.in(KnowledgeChunk::getDocId, docIds)`”，或新增 Mapper XML 使用参数绑定。
3. SSE 总超时要结合 Python 端已有总预算和 Java RestClient read-timeout，避免重复超时策略互相打架。

#### 推荐方案

先不做大拆分，先修两个具体风险：

1. `loadFallbackChunks()` 避免 `inSql()` 拼接  
   优先方案：注入 `KnowledgeDocMapper`，先查 PUBLISHED docIds，再用 `.in()` 查询 chunks。

2. SSE 总时长保护  
   在 `consumeSseStream()` 增加 `maxDuration`，建议 180s；超时后记录日志并返回失败结果，同时调用 `cancelTask(taskId)`。

之后再拆分职责：

```text
AgentRequestContextBuilder
  - buildConnectionConfig()
  - buildUserPermissionsMap()
  - buildConversationHistory()
  - loadFallbackChunks()

AgentSseResultConsumer
  - consumeSseStream()
  - handleProgressEvent()

PythonAgentClientImpl
  - executeAsync()
  - cancelTask()
```

#### 验收标准

- fallback chunks 查询不再拼接 SQL 字符串
- SSE 读取有总时长保护
- 查询任务结果回写、进度回写、助手消息保存行为不变
- 取消任务仍尽力通知 Python

### 4.2 `PythonHealthChecker` 共享可变健康状态对象

#### 当前状态

项目未满足。`PythonHealthChecker` 使用：

```java
private final AtomicReference<ServiceHealthInfo> healthInfo = new AtomicReference<>();
```

但 `ServiceHealthInfo` 是 `@Data` 可变对象，`checkHealth()` 中会直接修改同一个实例：

```java
current.setStatus(...);
current.setConsecutiveFailures(...);
current.setLastCheckTime(...);
current.setLastErrorMessage(...);
```

`AtomicReference` 只保证引用读写原子，不保证引用对象内部多个字段修改的原子性。调度线程更新多个字段时，其他线程调用 `getHealthInfo()` 可能看到半更新状态。

#### 推荐方案

将 `ServiceHealthInfo` 改为不可变快照对象。每次健康检查生成新对象，并通过 `healthInfo.set(newInfo)` 整体替换。

可选实现：

- 使用 Java `record`
- 或保留 class，但移除 setter，提供构造函数/静态工厂方法
- `getHealthInfo()` 返回当前不可变快照

#### 验收标准

- `PythonHealthChecker` 不再原地修改 `ServiceHealthInfo`
- `ServiceHealthInfo` 对外不可变
- 状态恢复通知和失败阈值逻辑不变

### 4.3 `PythonPoolClientImpl` 重复代码与状态异常不可见

#### 当前状态

项目未满足。`resetPool()` 与 `destroyPool()` 结构高度重复，且 `onStatus()` 中只记录日志，不抛出异常：

```java
.onStatus(HttpStatusCode::isError, (request, response) ->
        log.warn("...", response.getStatusCode()))
```

这与类注释“调用失败时仅记录警告日志，不抛出异常，保证主业务流程不受影响”一致，所以不是功能 Bug。但它有两个问题：

1. 重复代码增加维护成本。
2. HTTP 状态异常和真正调用成功都会继续走到 `log.info("已通知...")`，操作是否成功不可见。

#### 推荐方案

抽取公共方法，但保留 best-effort 语义：

```java
private boolean executePoolOperation(String operationName, Long datasourceId, Consumer<RestClient> call) {
    // 成功返回 true
    // HTTP error / ResourceAccessException / 其他异常返回 false，并记录 warn
}
```

或更简单地抽取：

```java
private boolean resetOrDestroyPool(Long datasourceId, String operationName, Supplier<RestClient.RequestHeadersSpec<?>> request)
```

关键是 `onStatus()` 中应标记失败，例如抛出内部异常后在外层 catch 统一 warn，或返回 false。Controller/Service 层如果仍不需要强失败，可以忽略返回值。

#### 验收标准

- `resetPool()` 与 `destroyPool()` 不再重复大段 RestClient 调用模板
- HTTP 4xx/5xx 不再被记录为“已通知成功”
- 主业务流程仍不因连接池重置/销毁失败而中断

### 5.5 `@Qualifier` 注解风格不一致

#### 当前状态

项目中存在两种 `@Qualifier` 注解风格：

**风格一：构造函数参数注入**
```java
// PythonRagClientImpl、PythonKnowledgeClientImpl、PythonPoolClientImpl
public PythonRagClientImpl(@Qualifier("pythonRestClient") RestClient restClient) {
    this.restClient = restClient;
}
```

**风格二：字段注解 + @RequiredArgsConstructor**
```java
// PythonAgentClientImpl
@RequiredArgsConstructor
public class PythonAgentClientImpl {
    @Qualifier("pythonRestClient")
    private final RestClient restClient;
}
```

**问题**：
1. 代码风格不一致，影响可读性
2. 风格二虽然功能正确（Lombok 会将 `@Qualifier` 传递到构造函数参数），但容易被误读为字段注入
3. 新开发者可能不清楚两种风格的区别

**推荐方案**：
- 统一使用风格一（构造函数参数注入），与项目中大多数类保持一致
- 或者在风格二的类中添加注释说明 `@Qualifier` 会通过 `@RequiredArgsConstructor` 传递到构造函数

**优先级**：P2（代码风格优化，不影响功能）

### 5.6 Python RestClient 连接复用与 `@Primary` 防误用

#### 当前状态

项目部分满足。`PythonRestClientConfig` 已经集中定义了 3 个 RestClient：

- `pythonRestClient`
- `pythonShortTimeoutRestClient`
- `pythonHealthRestClient`

但仍有两个可改进点：

1. 当前基于 `SimpleClientHttpRequestFactory`，通常不提供成熟 HTTP 连接池复用能力。频繁调用 Python 服务时，连接创建成本和并发能力不如 Apache HttpClient / JDK HttpClient 等池化方案。
2. `pythonRestClient` 标记了 `@Primary`。这能简化默认注入，但也可能让未来新增的 `RestClient` 注入点在忘记 `@Qualifier` 时意外拿到主 Python 客户端。

#### 推荐方案

此项不是当前功能 Bug，放 P2。

- 短期：保留 `@Primary`，但规定所有 Python 客户端必须显式 `@Qualifier`，避免隐式依赖。
- 中期：评估替换为支持连接复用的 request factory，并为主调用、短超时调用、健康检查分别配置连接池容量。
- 如果项目未来接入非 Python 的 RestClient，建议取消 `@Primary`，强制所有 RestClient 注入点显式指定 qualifier。

#### 验收标准

- 新增 RestClient 注入点都有明确 `@Qualifier`
- 连接池方案上线前，压测确认不会改变超时行为

## 八、回滚策略

### 8.1 第一阶段回滚方案

**回滚方式**：`git revert` 相关 commit

**影响范围**：
- JDBC URL 构建逻辑
- Python 超时处理逻辑
- 异步执行器配置

**回滚步骤**：
```bash
# 1. 查看第一阶段的 commit
git log --oneline --first-parent

# 2. revert 相关 commit
git revert <commit-hash>

# 3. 验证编译通过
cd backend/DataOcean && mvn compile

# 4. 验证测试通过
mvn test
```

**回滚时间**：< 5 分钟

**风险点**：
- 回滚后 `SchemaStatisticsServiceImpl` 会恢复无超时状态

## 十、风险与注意事项

| 项目 | 风险 | 缓解措施 | 注意事项 |
|---|---|---|---|
| JDBC URL 统一 | 中 | 先在测试环境验证超时参数 | 不要把连接测试改成 60s 慢超时 |
| Python 超时处理 | 低 | 充分测试超时和连接失败场景 | 区分读超时和连接不可用，不要扩大重试 |
| GlobalExceptionHandler | 低 | 修判断逻辑即可 | 先不改 HTTP 状态语义 |
| AiConfigController 拆分 | 中 | 逐步迁移，每步都测试 | 不改变 API、权限、密钥存储、reload best-effort |
| PythonAgentClientImpl | 中 | 先修风险点，再拆职责 | 拆分要小步走 |
| PermissionValidationSupport | 低 | 统一错误信息格式 | 不要把策略专属校验过度抽走 |
| CollectorContext | 低 | 先重构主采集 Collector | 统计采集另行处理 |
| ServiceHealthInfo | 低 | 改为不可变对象 | 保持状态恢复逻辑不变 |

## 十一、最终结论

这份方案里的问题在当前项目中大多数仍然存在，不能删除。需要做的是：

1. 把“已实施”的错觉去掉，明确当前基本都是待实施。
2. 保留所有漏洞/技术债，但按 P0/P1/P2 排序。
3. 修正原方案中不合理的实现细节，比如 private `@Retryable`、连接测试超时被统一成慢超时、`inSql()` 替换方式不成立、治理状态和审核状态混用。
4. 先做 P0 稳定性修复，再做 Controller/Client 分层，最后做低优先级清理。

本版文档可直接作为后续开发拆任务依据。
