# 文档修改要点

## 1. RAG 的数据来源必须先经过元数据治理

RAG 不能直接依赖人工随手写的 `skills.md`，否则知识来源不可信。正确链路应为：

1. 系统先全量读取数据库元数据，包括表、字段、主外键、索引、注释、枚举值候选、数据量级和变更记录。
2. AI 基于完整元数据先“认识”数据库，按业务域、事实表/维度表、关联路径、指标口径、废弃字段、敏感字段等维度生成结构化业务说明。
3. 生成内容进入 `skills.md` 草稿，由数据分析师或管理员审核、修正和确认。
4. 审核通过后的 `skills.md` 版本才允许切分、向量化并进入 RAG 召回。
5. RAG 返回结果必须能追溯到元数据版本和 `skills.md` 版本，避免不可解释、不可审计。

## 2. 数据治理是 NL2SQL 可信度的核心

数据治理不能只写成“废弃字段黑名单”或“查询失败后告警”，而要作为 RAG 的前置准入流程：

- 治理对象首先是元数据本身：表、字段、主外键、索引、注释、字段含义、指标单位、Join Path、枚举值、数据负责人。
- 正确性治理：校验元数据的完整性、准确性、一致性、时效性、可追溯性。
- 元数据治理：保证表字段信息完整、注释可用、归属清楚、状态明确。
- 语义治理：明确业务指标口径、Join 规则、事实表/维度表、枚举值解释。
- 准入治理：未审核、低可信、敏感、废弃、阻断状态的表字段不得进入 RAG。
- 血缘治理：记录每次查询使用的表字段，反向更新字段可信度和变更影响范围。
- 版本治理：`skills.md`、元数据、向量索引必须有版本关系，方便回滚和审计。

老师强调的核心可以概括为：

> 所有依据必须来自元数据；元数据必须先被治理证明是正确的；只有正确元数据生成的知识才能进入 RAG。

所以系统里需要有元数据质量规则、质量问题清单、负责人审核、元数据快照发布、`skills.md` 审核发布、RAG 准入门禁这条闭环。

## 3. 对主构想文档的调整方向

需要把 `docs/nl2sql-单库多表版-项目构想.md` 中的数据治理部分从“辅助模块”提升为主链路：

`数据库元数据采集 -> 元数据质量校验 -> 问题清单与人工确认 -> 发布元数据快照 -> 元数据画像 -> AI 生成 skills 草稿 -> 人工审核 -> 发布 skills 版本 -> 向量化 -> RAG 检索 -> SQL 生成 -> 血缘与反馈回流`

这样才能说明：系统不是让大模型凭空猜数据库，而是先把数据库转化为可信的业务知识层，再让 RAG 检索这层受治理的知识。
