# NL2SQL 单库多表项目功能点拆分说明

本文档基于 `docs/nl2sql-单库多表版-项目构想.md` 和 `docs/test.md` 整理，用于概括说明系统需要建设哪些功能模块、每个模块解决什么问题，以及各模块之间的关系。

## 1. 项目核心目标

本项目不是单纯做一个“自然语言转 SQL”工具，而是建设一个面向企业内部的智能数据查询与治理平台。

核心目标是：

- 让业务人员通过自然语言查询单个业务库中的多张表。
- 让 AI 生成 SQL 时有可信依据，而不是凭空猜表、猜字段、猜口径。
- 通过元数据治理、`skills.md`、字段 Tag、可信度、RAG、血缘和审计，形成可解释、可追溯、可持续维护的查询闭环。

项目 MVP 范围限定为：

- 支持多个 MySQL 数据源接入。
- 用户每次查询只能选择一个数据源。
- 在单个数据库内支持多表关联查询。
- 暂不做跨库联邦查询。

## 2. 系统主流程

系统整体链路可以概括为：

```text
数据源接入
  -> 元数据采集
  -> 元数据质量校验
  -> 元数据问题治理
  -> 发布可信元数据快照
  -> AI 生成 skills.md 草稿
  -> 人工审核发布 skills.md
  -> 向量化进入 RAG
  -> 用户自然语言提问
  -> RAG 召回相关表字段和业务知识
  -> AI 生成 SQL
  -> SQL 安全校验与沙箱执行
  -> 返回表格、图表、口径说明
  -> 查询血缘、审计、反馈回流
```

这个流程强调一点：**所有 AI 使用的依据必须来自元数据，元数据必须先经过治理确认。**

## 3. 功能模块总览

| 模块 | 主要功能 | 解决的问题 |
|------|----------|------------|
| 数据源管理 | 接入 MySQL 数据库、配置只读账号、测试连接、管理访问权限 | 让系统安全连接企业数据库 |
| 元数据采集 | 读取表、字段、索引、主外键、注释、数据量级、字段样本统计 | 建立 AI 可理解数据库的基础材料 |
| 元数据治理 | 校验元数据完整性、准确性、一致性、时效性、可追溯性 | 确保 AI 的依据是正确的 |
| 元数据版本管理 | 生成元数据快照、记录变更、审核发布、旧版本失效 | 保证知识来源可追溯、可回滚 |
| `skills.md` 管理 | 基于元数据生成草稿、人工审核、版本管理、向量化 | 把数据库结构转成业务语义知识 |
| Schema RAG | 按用户问题召回相关表、字段、业务口径、SQL 模板 | 避免把全库 Schema 一次性塞给大模型 |
| NL2SQL Agent | 意图理解、SQL 生成、错误重试、结果解释、图表推荐 | 完成自然语言到可执行查询的转换 |
| SQL 安全与沙箱 | 只允许 SELECT、限制行数、超时控制、敏感字段脱敏 | 防止误删、越权、慢查询和敏感数据泄露 |
| 字段 Tag 与可信度 | 标记字段类型、治理状态、可信分数、用户反馈 | 指导 AI 优先使用高可信字段 |
| 血缘与审计 | 记录每次查询用到的表字段、SQL、用户、结果状态 | 支持追溯、告警、知识修正和合规审计 |
| 前端问答端 | 数据源选择、自然语言提问、表格图表展示、反馈 | 面向普通员工提供查询入口 |
| 后台治理端 | 数据源、元数据、`skills.md`、Prompt、可信度、审计管理 | 面向管理员和分析师维护系统可信度 |
| 可观测与运维 | 日志、指标、链路追踪、告警、备份、Docker Compose | 保证系统可部署、可监控、可维护 |

大模型选择：为什么选择通用大模型而不是专用的nl5sql模型

## 4. 数据源管理功能

数据源管理负责接入企业内部业务数据库。

需要包含的功能点：

- 新增、编辑、删除数据源。
- 配置数据库类型、JDBC URL、主机、端口、库名。
- 配置只读账号和加密后的数据库密码。
- 测试数据库连通性。
- 管理数据源启用、禁用状态。
- 为用户、角色、部门配置数据源访问权限。
- 限制每次查询只能在当前选中的单个数据源内执行。

核心要求：

- 业务库连接必须使用只读账号。
- 数据库密码不能明文存储。
- 用户没有权限的数据源不能出现在前台查询入口。

## 5. 元数据采集功能

元数据采集是整个系统的基础。AI 不能直接凭用户问题猜数据库，而要先依赖系统采集到的元数据。

需要采集的内容：

- 表名、表注释、表类型。
- 字段名、字段类型、字段注释、是否可空。
- 主键、外键、索引。
- 表数据量级。
- 字段空值率、枚举值 TopN、最大最小值、日期范围等轻量统计。
- 历史 SQL 中出现过的表、字段、Join Path。
- BI 报表中已经验证过的指标口径和高频字段。

采集结果需要形成 `metadata_snapshot` 元数据快照，作为后续治理、`skills.md` 生成和 RAG 向量化的版本依据。

## 6. 元数据治理功能

元数据治理是本项目的关键。这里治理的不是查询结果，而是治理 AI 使用的一切依据。

元数据治理需要做五类校验：

| 治理维度 | 说明 |
|----------|------|
| 完整性 | 表、字段、主键、索引、注释是否采集完整 |
| 准确性 | 字段含义、表用途、指标口径是否真实可信 |
| 一致性 | 同名字段、同类指标、Join Path 是否存在冲突 |
| 时效性 | 元数据是否反映当前数据库状态 |
| 可追溯性 | 每条知识是否能追溯到元数据快照、审核人和版本 |

需要包含的功能点：

- 元数据质量规则管理。
- 自动执行质量校验。
- 生成元数据质量分。
- 生成元数据问题清单。
- 将问题分派给数据负责人或分析师。
- 支持问题处理、确认、驳回、关闭。
- 标记表字段治理状态：`DISCOVERED`、`NORMAL`、`RECOMMENDED`、`DEPRECATED`、`SENSITIVE`、`BLOCKED`。
- 只有审核通过、状态正常的元数据才能进入 `skills.md` 和 RAG。

典型问题包括：

- 表没有注释。
- 字段没有注释。
- 字段注释和样本数据不一致。
- 金额字段没有单位。
- 状态字段没有枚举说明。
- 同名字段在不同表中含义不同。
- 一个指标存在多套计算口径。
- Join Path 置信度低。
- 字段疑似废弃或敏感。

## 7. 元数据版本与审核功能

元数据不是一次性静态内容，数据库结构会持续变化，所以需要版本控制。

需要包含的功能点：

- 每次 Schema 同步生成一个元数据快照。
- 记录快照编号、Schema Hash、表数量、字段数量、质量分。
- 对比新旧快照差异，包括新增表、删除表、字段类型变化、字段注释变化、索引变化。
- 对存在高风险问题的快照禁止发布。
- 只有 `PUBLISHED` 状态的元数据快照才能用于生成 `skills.md` 和 RAG 向量。
- 旧快照在数据库变更后可以标记为 `EXPIRED`。

元数据快照状态建议包括：

- `DRAFT`
- `CHECKING`
- `ISSUE_FOUND`
- `APPROVED`
- `PUBLISHED`
- `EXPIRED`

## 8. `skills.md` 业务知识库功能

`skills.md` 是业务语义说明书，但它不能从空白文档随手编写，而应该由元数据治理结果生成草稿，再由人审核。

需要包含的功能点：

- 基于已发布元数据快照生成 `skills.md` 草稿。
- 自动填充数据库基本信息、核心表说明、Join Path、指标候选、枚举解释、字段防坑。
- 分析师补充业务流程、指标口径和常见 SQL 模板。
- 支持 Markdown 在线编辑。
- 支持草稿、待审核、审核通过、已发布、已废弃状态。
- 支持版本对比和回滚。
- 每个版本绑定 `metadata_snapshot_id`。
- 发布前校验文档中引用的表字段是否存在、是否已审核、是否被阻断。

`skills.md` 主要记录：

- 数据库所属业务域。
- 核心业务流程。
- 事实表、维度表、流水表、配置表。
- 关键 Join Path。
- 指标口径，例如 GMV、订单数、退款率。
- 字段防坑，例如废弃字段、敏感字段、易混字段。
- 枚举值解释。
- 高频问题和 Few-shot SQL 模板。

## 9. RAG 召回功能

RAG 负责从向量库中召回和用户问题最相关的表、字段和业务知识。

需要包含的功能点：

- 将已审核元数据、字段说明、`skills.md` 切片向量化。
- 向量写入 Milvus。
- 每条向量携带 `datasource_id`、`metadata_snapshot_id`、`knowledge_version_no`、`review_status`、`governance_status`。
- 用户提问时只在当前选定数据源范围内检索。
- 强制过滤未审核、废弃、敏感、阻断状态的内容。
- 支持语义召回、关键词召回、模板召回。
- 对召回结果进行规则加权和重排序。
- 返回 Top 5-10 张相关表及关键字段。

RAG 的目标不是“召回越多越好”，而是精准召回当前问题需要的可信上下文。

## 10. NL2SQL Agent 功能

NL2SQL Agent 负责把用户自然语言转成安全可执行的 SQL。

需要包含的功能点：

- 接收用户问题、当前数据源、权限范围、历史对话摘要。
- 调用 RAG 召回相关 Schema 和业务知识。
- 根据 Prompt 模板生成 SQL。
- 结合字段可信度优先使用高可信字段。
- 对 SQL 执行安全校验。
- SQL 执行失败时读取错误信息并自动重试。
- 最多重试固定次数，避免无限循环。
- 执行成功后生成结果解释。
- 根据数据结构推荐 ECharts 图表。

Agent 可以拆成以下节点：

- `Schema_Retriever_Node`
- `SQL_Generator_Node`
- `SQL_Validator_Node`
- `SQL_Executor_Node`
- `Data_Visualizer_Node`

## 11. SQL 安全与执行沙箱功能

AI 生成的 SQL 不能直接执行，必须经过安全网关。

需要包含的功能点：

- 只允许 `SELECT` 查询。
- 禁止 `INSERT`、`UPDATE`、`DELETE`、`DROP`、`ALTER` 等语句。
- 禁止多语句执行。
- 禁止访问未授权表和字段。
- 自动追加行数限制。
- 控制查询超时时间。
- 限制子查询嵌套深度。
- 限制连接池数量。
- 使用只读事务执行。
- 对敏感字段执行脱敏。

这个模块解决的是安全和稳定性问题，防止 AI 生成危险 SQL 或拖垮数据库。

## 12. 字段 Tag 与可信度功能

字段 Tag 和可信度用于告诉 AI 哪些字段更可靠，哪些字段需要谨慎使用。

需要包含的功能点：

- 为字段打业务标签，例如金额、时间、状态、用户 ID、敏感字段。
- 标记字段是否推荐、废弃、敏感、阻断。
- 为字段维护 0-100 的可信度分数。
- 根据 `skills.md`、人工审核、BI 报表、历史成功查询初始化可信度。
- 根据用户反馈和查询结果动态调整可信度。
- 普通用户反馈不能直接改变可信度，必须进入审核队列。

可信度对 SQL 生成的影响：

- 高可信字段优先使用。
- 中可信字段可以使用，但要结合注释和业务知识判断。
- 低可信字段谨慎使用。
- 废弃、敏感、阻断字段默认不能被 RAG 召回。

## 13. 血缘与审计功能

血缘和审计负责回答“这次查询依据了什么、用了哪些表字段、结果从哪里来”。

需要包含的功能点：

- 记录用户问题。
- 记录生成的 SQL。
- 记录使用的表和字段。
- 记录依据的 `skills.md` 版本和元数据快照。
- 记录 SQL 执行状态、耗时、返回行数。
- 解析 SQL 生成表级血缘和字段级血缘。
- 当表字段变更时，反查受影响的查询、知识片段和模板。
- 支持查询日志检索和导出。

血缘需要和数据库元数据绑定，不能只是独立的 SQL 日志。否则后续字段变更、口径变更时无法判断影响范围。

## 14. 前端问答端功能

前端问答端面向普通业务用户。

需要包含的功能点：

- 数据源选择器。
- 自然语言输入框。
- 多轮对话展示。
- 查询进度展示。
- SQL 可选展示。
- 表格结果展示。
- ECharts 图表展示。
- 图表类型切换。
- 结果导出。
- 口径说明和溯源面板。
- 用户反馈：赞、踩、原因选择。
- 历史会话查看和删除。

前端不需要暴露复杂治理概念，但需要让用户知道本次结果依据了哪些表、字段和业务规则。

## 15. 后台治理管理端功能

后台治理端面向管理员、数据分析师、数据管理员和安全管理员。

需要包含的功能点：

- 数据源管理。
- 元数据同步和快照管理。
- 元数据质量看板。
- 元数据问题处理。
- 表字段治理状态维护。
- `skills.md` 草稿生成、编辑、审核、发布。
- Prompt 模板管理。
- 字段 Tag 和可信度管理。
- 用户反馈审核。
- 查询审计查看。
- 血缘图谱查看。
- 敏感字段和脱敏规则管理。
- 成本和 Token 用量监控。

后台治理端是系统可信度的核心入口。

## 16. Prompt 管理功能

Prompt 不应该硬编码在代码中，而应该统一管理。

需要包含的功能点：

- Prompt 模板数据库存储。
- Prompt 版本管理。
- Prompt 修改记录。
- Prompt 回滚。
- Token 预算控制。
- 不同环节使用不同 Prompt，例如 SQL 生成、SQL 修复、图表生成、结果解释。
- Prompt 中注入当前数据源、已召回 Schema、`skills.md`、字段可信度、历史错误信息。

Prompt 管理的目标是让 AI 行为可调试、可回滚、可审计。

## 17. 权限与安全功能

权限体系需要覆盖功能权限、数据权限和结果权限。

需要包含的功能点：

- 用户、角色、部门管理。
- 功能权限控制。
- 数据源级访问控制。
- 表级访问控制。
- 字段级访问控制。
- 行级过滤策略。
- 结果导出权限。
- SQL 查看权限。
- 敏感字段脱敏。
- 审计日志留存。

安全边界：

- 用户不能查询未授权数据源。
- 被标记为 `BLOCKED`、`DEPRECATED` 的表字段不能进入 RAG，也不能执行。
- 敏感字段返回前必须脱敏。

## 18. 图表生成与结果解释功能

系统不仅返回 SQL 查询结果，还需要把结果转化为业务人员能理解的形式。

需要包含的功能点：

- 自动判断适合的图表类型。
- 生成 ECharts 配置。
- 支持柱状图、折线图、饼图、表格。
- 图表导出。
- 生成口径说明。
- 展示本次查询使用的表、字段、指标定义和可信度。

这部分提升的是可用性和可解释性。

## 19. 错误处理与降级功能

NL2SQL 链路涉及数据库、向量库、LLM、Python 服务和 Java 网关，必须有降级策略。

需要包含的功能点：

- 数据源连接失败提示。
- RAG 召回失败降级。
- LLM 调用失败重试。
- SQL 执行失败后自动修复。
- 超时后返回友好提示。
- Python 服务不可用时 Java 网关告警。
- SSE 中断后支持重连或轮询恢复。

错误处理要避免用户只看到“系统异常”，而应该能知道失败发生在哪个环节。

## 20. 可观测性与运维功能

系统需要可部署、可监控、可维护。

需要包含的功能点：

- 结构化日志。
- Metrics 指标。
- 链路追踪。
- Python 服务健康检查。
- SQL 执行耗时监控。
- LLM 调用次数和 Token 费用统计。
- 慢查询告警。
- 向量化任务状态监控。
- 管理库备份。
- Docker Compose 部署。
- Flyway 管理数据库迁移。

## 21. MVP 优先级建议

MVP 阶段建议优先完成以下能力：

| 优先级 | 功能点 | 原因 |
|--------|--------|------|
| P0 | 数据源接入与只读连接 | 没有数据源无法查询 |
| P0 | 元数据采集与快照 | 所有依据必须来自元数据 |
| P0 | 元数据质量校验与审核发布 | 防止错误元数据进入 RAG |
| P0 | `skills.md` 草稿生成与审核 | 补足 Schema 缺少业务语义的问题 |
| P0 | RAG 向量化与数据源隔离召回 | 解决大库多表上下文过载问题 |
| P0 | SQL 生成、校验、执行 | 完成核心查询链路 |
| P0 | SQL 安全沙箱 | 防止危险 SQL 和慢查询 |
| P1 | 表格和图表展示 | 提升演示和使用体验 |
| P1 | 字段 Tag 与可信度 | 提高 SQL 生成稳定性 |
| P1 | 查询审计和基础血缘 | 支持可解释和问题追踪 |
| P2 | 成本监控、长期记忆、复杂血缘图谱 | 企业化增强能力 |

## 22. 总结

这个项目可以拆成三条主线：

1. **可信数据依据线**：数据源接入 -> 元数据采集 -> 元数据治理 -> 元数据快照 -> `skills.md` -> RAG。
2. **智能查询执行线**：用户提问 -> RAG 召回 -> SQL 生成 -> 安全校验 -> 执行 -> 图表和解释。
3. **治理反馈闭环线**：查询审计 -> 字段可信度 -> 用户反馈 -> 血缘分析 -> 元数据和知识库修正。

其中最重要的是第一条线。只有元数据被治理正确，后面的 `skills.md`、RAG、SQL 生成和结果解释才有可信基础。
