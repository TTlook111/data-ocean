### 1. 时间计划解析存在逻辑漏洞（安全风险）

**文件**: `PermissionCalculatorImpl.java:305-312`

**问题代码**:
```java
String weekdaysStr = schedule.substring(schedule.indexOf("\"weekdays\"") + 11);
weekdaysStr = weekdaysStr.substring(0, weekdaysStr.indexOf("]"));
boolean dayAllowed = weekdaysStr.contains(String.valueOf(dayOfWeek));
```

**问题描述**:
- 使用 `String.contains()` 检查工作日会导致误匹配
- 当 `dayOfWeek=1` 时，会匹配到 `"1"`, `"10"`, `"11"`, `"12"` 等
- 例如：weekdays=[10,11,12] 时，周一(dayOfWeek=1)会被错误允许

**复现场景**:
```json
{
  "weekdays": [10, 11, 12],
  "hours": {"from": "09:00", "to": "18:00"}
}
```
- 周一（dayOfWeek=1）会被错误允许访问
- 周二（dayOfWeek=2）也会被错误允许访问

**修复建议**:
```java
// 方案1：使用 JSON 解析器（推荐）
ObjectMapper objectMapper = new ObjectMapper();
JsonNode weekdaysNode = objectMapper.readTree(schedule).path("weekdays");
if (weekdaysNode.isArray()) {
    boolean dayAllowed = false;
    for (JsonNode node : weekdaysNode) {
        if (node.asInt() == dayOfWeek) {
            dayAllowed = true;
            break;
        }
    }
    if (!dayAllowed) return false;
}

// 方案2：使用正则表达式精确匹配
Pattern pattern = Pattern.compile("\\b" + dayOfWeek + "\\b");
boolean dayAllowed = pattern.matcher(weekdaysStr).find();
```

**影响范围**: 权限系统时间条件判断，可能导致非授权时间访问  
**严重程度**: 🔴 高（安全漏洞）  
**修复优先级**: 立即

### 2. 部门权限合并逻辑顺序错误（功能缺陷）

**文件**: `DatasourceAccessServiceImpl.java:199-201`

**问题代码**:
```java
applyDepartmentGrant(decision, grants, deptIds);  // 如果命中会直接 return
applyRoleGrants(decision, grants, roleIds);        // 可能被跳过
applyUserGrant(decision, grants, userId);          // 可能被跳过
```

**问题描述**:
- `applyDepartmentGrant` 方法在找到部门授权后会直接 return（第217行）
- 这导致角色授权和用户授权永远不会被应用
- 违背了"并集语义"的设计原则

**复现场景**:
```
用户A 属于部门D，有以下授权：
1. 部门D对数据源X有 ALLOW 权限
2. 用户A对数据源X有 DENY 权限（应拒绝访问）

预期结果：用户A被拒绝访问（用户权限优先）
实际结果：用户A被允许访问（部门权限覆盖了用户权限）
```

**修复建议**:
```java
// 方案1：调整优先级顺序（用户 > 角色 > 部门）
applyDepartmentGrant(decision, grants, deptIds);
applyRoleGrants(decision, grants, roleIds);
applyUserGrant(decision, grants, userId);  // 用户授权最后应用，覆盖之前的结果

// 方案2：合并所有维度的权限（并集语义）
boolean canQuery = false;
boolean canExport = false;
boolean canViewSql = false;
String accessEffect = EFFECT_DENY;  // 默认拒绝

for (DatasourceAccess grant : grants) {
    if (isDeny(grant)) {
        // DENY 优先，直接返回
        decision.setCanQuery(false);
        decision.setCanExport(false);
        decision.setCanViewSql(false);
        decision.setAccessEffect(EFFECT_DENY);
        return decision;
    }
    // ALLOW 时取并集
    canQuery |= Boolean.TRUE.equals(grant.getCanQuery());
    canExport |= Boolean.TRUE.equals(grant.getCanExport());
    canViewSql |= Boolean.TRUE.equals(grant.getCanViewSql());
    accessEffect = EFFECT_ALLOW;
}

decision.setCanQuery(canQuery);
decision.setCanExport(canExport);
decision.setCanViewSql(canViewSql);
decision.setAccessEffect(accessEffect);
```

**影响范围**: 多维度权限合并，可能导致权限丢失或权限提升  
**严重程度**: 🔴 高（功能缺陷）  
**修复优先级**: 尽快

### 3. 权限缓存失效策略过于激进（性能问题）

**文件**: `PermissionCalculatorImpl.java:82-91`

**问题代码**:
```java
if (datasourceId == null) {
    cache.clear();  // 清除所有缓存
    return;
}
```

**问题描述**:
- 当 `datasourceId` 为 null 时，清除整个权限缓存
- 这会导致大量不必要的权限重新计算
- 在高并发场景下可能造成性能抖动

**复现场景**:
```
场景：角色权限变更（影响该角色下所有用户）
触发：PermissionChangedEvent(subjectId=角色ID, datasourceId=null)
结果：清除所有权限缓存（包括无关用户和数据源的缓存）
```

**修复建议**:
```java
if (datasourceId == null) {
    // 按 subjectId 清除相关缓存，而不是全部清除
    String prefix = subjectId + ":";
    cache.entrySet().removeIf(e -> e.getKey().startsWith(prefix));
    log.debug("Permission cache invalidated for subjectId={}", subjectId);
    return;
}
```

**影响范围**: 系统性能，高并发时可能导致响应延迟  
**严重程度**: 🔴 高（性能问题）  
**修复优先级**: 计划修复

## 🧭 产品级补充审查：权限、信息架构与后台业务规范

> 补充日期：2026-06-24  
> 补充背景：开发者实际使用后台后反馈“页面分散、层级混乱、像玩具项目”，并指出权限设计不应停留在“参考大公司”，而应回答真实组织里不同人员如何分配权限、如何协作。

### 1. 当前后台的核心问题不是视觉，而是信息架构按技术模块平铺

当前后台侧边栏把能力拆成了很多技术入口：用户、角色、部门、数据源、同步任务、快照、表浏览器、质量看板、问题清单、治理状态、生命周期、版本历史、知识库、Skills 编辑器、知识审核、Prompt、字段标签、可信度、反馈审核、审计、慢查询、血缘、访问控制、策略编辑器、系统健康、操作日志、AI 配置等。

这些页面本身都有价值，但组织方式更像“把数据库表和后端模块暴露给管理员”，而不是“让不同岗位完成自己的工作”。真实企业项目通常不会让所有后台用户面对同一组扁平菜单，而是围绕岗位、对象和流程组织：

- 谁负责接入数据？
- 谁负责确认数据含义？
- 谁负责审批发布？
- 谁负责给业务用户授权？
- 谁负责排查查询失败、慢查询和安全审计？
- 普通业务用户在哪里申请权限、查看申请进度？

如果后台不能回答这些问题，即使页面数量很多，也会显得割裂。

### 2. 权限设计应从“岗位职责”出发，而不是只从权限码出发

当前系统已经有 `USER / ROLE / DEPARTMENT` 三种授权主体，也有数据源级、行列级、脱敏、导出、查看 SQL 等能力。但还缺一层“人怎么分配”的产品定义。

建议把权限模型拆成三类：

| 类型 | 解决的问题 | 示例 |
| --- | --- | --- |
| 功能权限 | 这个人能进哪些后台页面、能执行哪些按钮操作 | `datasource:manage`、`metadata:manage`、`audit:view` |
| 数据权限 | 这个人能查哪些数据源、表、字段、行 | 数据源授权、表级状态、列禁用、行过滤 |
| 流程权限 | 这个人在治理流程中能提交、审核、发布、回滚什么 | 知识审核、快照发布、权限审批、Prompt 审批 |

企业项目里真正重要的是“岗位包”，而不是让管理员逐个理解底层权限点。建议预设以下角色，并把页面、操作和数据范围绑定到角色说明里：

| 角色 | 典型人员 | 主要职责 | 默认可见入口 |
| --- | --- | --- | --- |
| 业务查询用户 | 业务人员、运营、销售主管 | 使用自然语言查询已授权数据，提交反馈，申请临时权限 | 智能查询、我的会话、我的权限申请 |
| 数据分析师 | BI、数据分析、指标负责人 | 维护业务指标、术语、查询场景，审核字段反馈 | 语义资产、术语、指标/Skills、反馈审核、查询效果 |
| 数据管理员 | 数据平台/DBA/数据开发 | 接入数据源、同步元数据、处理采集异常、维护元数据质量 | 数据资产、数据源、采集任务、表字段治理、快照 |
| 数据安全管理员 | 安全/合规/数据负责人 | 配置授权、脱敏、行列级策略，处理临时访问审批 | 权限中心、访问策略、访问申请、审计 |
| 治理负责人 | 数据治理负责人、项目管理员 | 关注整体治理进度、发布状态、阻塞事项和跨团队协同 | 治理工作台、发布中心、质量趋势、待办 |
| 系统运维管理员 | 运维、平台管理员 | 维护服务健康、AI 配置、告警、操作日志 | 系统运维、服务健康、AI 配置、告警/配额 |
| 超级管理员 | 项目初始化/最高权限账号 | 系统兜底配置、角色权限模板维护 | 全部入口，但日常不应作为业务操作账号 |

权限分配原则：

- 角色用于分配岗位能力，部门用于默认数据范围，用户授权只用于例外补充或临时处理。
- `DENY` 要优先于 `ALLOW`，用户级例外要能覆盖部门/角色默认授权。
- 普通业务用户不应该看到后台入口；数据分析师也不应该看到系统健康、AI 密钥、全局用户管理等运维入口。
- 数据源授权、行列级策略、脱敏策略、导出权限、SQL 可见性要放在同一条“数据访问决策”里展示，否则管理员很难判断一个人到底能看到什么。
- 权限审批应该成为普通用户的正向入口：当查询缺少权限时，系统提示“申请访问”，而不是只显示失败。

### 3. 后台应从“页面集合”改成“业务工作台 + 对象详情 + 流程待办”

建议把当前后台重组为 6 个一级域，每个一级域下面再承载页面。一级菜单不宜超过 6-7 个，否则用户无法建立稳定心智。

| 一级域 | 面向问题 | 合并/承载当前页面 |
| --- | --- | --- |
| 工作台 | 我今天要处理什么、系统是否健康、哪些数据源/知识待发布 | 概览、待办、通知、质量摘要、发布状态、最近失败任务 |
| 数据资产 | 这个数据源有什么表字段、采集是否正常、版本是什么 | 数据源管理、同步任务、同步调度、表浏览器、目录搜索、快照列表、版本历史 |
| 治理工作台 | 哪些表字段有质量问题、状态如何流转、发布前还差什么 | 质量看板、问题清单、治理状态、快照生命周期、变更事件 |
| 语义资产 | 业务术语、指标、Join Path、Skills 文档如何生产和发布 | 术语管理、知识库总览、Skills 编辑器、知识审核、Prompt 管理 |
| 权限与合规 | 谁能看什么、为什么能看、是否审批、是否脱敏 | 访问控制、策略编辑器、访问申请、权限变更记录、审计日志、血缘 |
| 系统运维 | 平台本身是否稳定、AI/告警/配额如何配置 | 服务健康、AI 配置、操作日志、慢查询、告警规则、配额管理 |

当前菜单的主要割裂点：

1. “元数据管理 / 元数据治理 / 版本管理”被拆成三个一级组，但实际用户会把它们理解为同一条数据资产生命周期。
2. “术语管理”放在知识库管理里可以，但路由又属于 `glossary`，权限却复用 `metadata:manage`，容易造成职责不清。
3. “字段治理”与“元数据治理”割裂，但字段标签、可信度、反馈审核本质上是字段治理闭环，应进入治理工作台或语义资产。
4. “审计管理”同时包含审计日志、慢查询、血缘，其中慢查询更偏系统运维，血缘更偏数据资产/合规，建议按任务重归类。
5. “AI 调优”只有 Prompt 管理一个入口，显得孤立；Prompt 与 Skills、术语、指标都属于语义资产和生成质量治理。

### 4. 推荐的业务闭环：让用户按流程走，而不是按模块找页面

DataOcean 的核心业务不是“管理很多表”，而是把数据库变成可被安全自然语言查询的数据产品。后台应该显式呈现这条链路：

```text
接入数据源
  -> 采集元数据
  -> 识别质量问题/敏感字段/关系
  -> 修复并审核治理状态
  -> 发布快照
  -> 生成/编辑/审核 skills.md
  -> 向量化发布
  -> 分配数据访问权限
  -> 业务用户查询
  -> 查询反馈/审计/血缘/质量回流
```

建议把后台首页从“功能快捷入口”改为“治理任务驾驶舱”：

- 今日待办：待审核快照、待审核知识、待处理质量问题、待审批访问申请、待审核字段反馈。
- 数据源状态：已接入、采集失败、待治理、已发布、知识向量化失败。
- 查询质量：成功率、失败原因 Top、低可信字段、用户差评聚合。
- 安全态势：高风险访问、临时授权即将过期、敏感字段查询、导出次数。
- 快捷动作：接入数据源、处理问题、发布知识、审批访问，而不是单纯跳转到所有页面。

### 4.1 用数据源生命周期判断“当前是否可以询问”

这个方向应作为后台重构的主线之一：每个数据源不只是一个连接配置，而是一个有生命周期的数据产品。系统应能明确告诉管理员和业务用户：这个数据源当前走到哪一步、还缺什么、是否已经允许自然语言询问。

建议为数据源建立统一的“可询问状态”：

| 阶段 | 状态 | 含义 | 是否允许询问 |
| --- | --- | --- | --- |
| 1 | 已接入 | 数据源连接已创建，连通性通过 | 否 |
| 2 | 已采集 | 表、字段、主键、索引等元数据已采集 | 否 |
| 3 | 待治理 | 发现缺失描述、敏感字段、关系不明、质量问题 | 否 |
| 4 | 治理中 | 数据管理员/分析师正在补充字段说明、Join Path、指标、术语 | 否 |
| 5 | 待审核 | 治理结果、快照或知识文档等待负责人审核 | 否 |
| 6 | 已发布快照 | 元数据快照已通过并发布 | 视知识状态而定 |
| 7 | 知识已发布 | skills.md 已审核、向量化成功、RAG 可召回 | 可以进入试运行 |
| 8 | 权限已配置 | 至少有角色/部门/用户获得查询权限，行列级策略已生效 | 可以正式询问 |
| 9 | 已下线/阻塞 | 数据源被禁用、治理状态 BLOCKED、知识失效或权限撤销 | 否 |

前台查询页不应该只判断“用户有没有数据源权限”，还要判断数据源是否处于可询问状态：

```text
可询问 = 数据源启用
      + 元数据快照已发布
      + skills.md 已发布且向量化成功
      + 不存在阻塞级治理问题
      + 当前用户有数据源查询权限
      + 当前时间满足权限策略
```

如果不可询问，页面应该给出明确原因，而不是只隐藏数据源或返回失败：

- “该数据源尚未完成元数据采集，请联系数据管理员。”
- “该数据源存在阻塞级治理问题，暂不可查询。”
- “该数据源知识库正在发布，完成后可询问。”
- “你尚未获得该数据源权限，可提交访问申请。”
- “当前时间不在授权访问时段内。”

后台数据源详情页应展示一个生命周期进度条：

```text
接入 -> 采集 -> 治理 -> 审核 -> 发布快照 -> 发布知识 -> 配置权限 -> 可询问
```

每一步都应显示：

- 当前状态：未开始、进行中、失败、待审核、已完成。
- 责任角色：数据管理员、数据分析师、治理负责人、安全管理员。
- 阻塞原因：失败日志、缺失字段说明、未审核问题、向量化失败、未配置权限。
- 下一步动作：重新采集、处理问题、提交审核、发布知识、配置权限、申请访问。

这样后台会从“页面集合”变成“数据源上线流程”。用户进入后台后不需要知道应该先点同步任务、再点快照、再点知识库、再点权限；系统直接告诉他当前卡在哪一步。

这个设计也能解决前台体验问题：业务用户只看到真正可用的数据源；不可用的数据源如果与自己相关，则显示可理解的状态和申请入口，而不是让用户误以为系统坏了。

### 5. 页面重设计不是换颜色，而是合并同类任务和增加上下文

建议优先重做以下页面/模块的层级：

#### 5.1 数据资产中心

将“数据源管理、同步任务、表浏览器、目录搜索、快照列表、版本历史”收束为一个数据源详情页。

推荐结构：

- 左侧：数据源列表/搜索。
- 右侧：数据源详情，使用 Tabs 承载“概览、表字段、采集任务、质量问题、快照版本、权限摘要、血缘”。
- 这样管理员围绕一个数据源完成所有动作，不需要在多个菜单之间来回跳。

#### 5.2 治理工作台

将“质量看板、问题清单、治理状态、字段可信度、反馈审核”合并为治理工作台。

推荐结构：

- 默认打开“待处理队列”，按严重程度、数据源、责任人过滤。
- 每个问题点进详情后能看到字段信息、历史查询、用户反馈、建议治理动作。
- 状态流转在详情页完成：待处理、修复中、待审核、已通过、已阻塞、已废弃。

#### 5.3 语义资产中心

将“术语、指标、Join Path、Skills、Prompt”放到同一业务域。

推荐结构：

- 术语和指标是业务语义输入。
- Join Path 和字段说明是 SQL 生成约束。
- Skills 文档是可审核、可发布的语义产物。
- Prompt 是系统级生成策略，不应和业务术语混为一个列表，但可以在同一域下作为高级配置。

#### 5.4 权限中心

将“访问控制、策略编辑器、访问申请、权限变更记录、审计日志”围绕“人能看什么”和“数据被谁看了”组织。

推荐结构：

- 按人员查看：某用户拥有哪些数据源、表、列、行过滤、脱敏、导出权限。
- 按数据源查看：某数据源授权给了哪些部门/角色/用户。
- 按申请查看：待审批、已批准、即将过期、已拒绝。
- 按风险查看：敏感字段访问、越权拦截、临时授权、导出行为。

#### 5.5 运维与平台设置

将“系统健康、AI 配置、操作日志、慢查询、告警、配额”合并为平台运维域。

这里的用户是平台管理员，不是数据治理人员。它应该默认隐藏给数据分析师和数据管理员。

#### 5.6 数据源、表、字段选择应统一为级联选择器

当前前端存在同一类对象在不同页面交互不一致的问题：有些页面已经能通过快照选择表和字段，例如字段标签页面；但有些页面仍要求手动输入表名或字段名，例如血缘查看、治理问题筛选、访问策略编辑器。

这会带来几个问题：

- 用户不知道真实表名/字段名，只能去别的页面复制。
- 手输容易拼错，导致查不到数据或配置错误策略。
- 页面之间没有继承上下文，用户刚选过数据源，到另一个页面还要重新选择。
- 企业项目的可信感下降，因为核心对象不像被系统正式管理，而像临时填字符串。

建议建立统一的资源选择链：

```text
数据源 -> 已发布快照 -> 表 -> 字段
```

不同页面按需要截取链路：

| 页面/场景 | 当前问题 | 建议 |
| --- | --- | --- |
| 血缘查看 | 表名、字段名手输 | 选择数据源后加载表列表；选择表后加载字段列表；字段可选 |
| 治理问题列表 | 表名筛选手输 | 改为可搜索表选择器，保留“全部表” |
| 访问策略编辑器 | 表名、列名手输 | 先选数据源，再选表，再选字段；表级策略时字段留空但通过开关表达 |
| 行级过滤配置 | 过滤表达式完全手写 | 至少提供字段选择、操作符选择和值输入；高级模式再允许手写表达式 |
| 知识生成 | 数据源和快照已可选 | 进一步展示该快照是否已审核、是否适合生成知识 |
| 字段标签 | 已有快照、表选择 | 可作为统一选择器模式的参考 |

前端应沉淀一个通用组件，而不是每个页面各自实现：

```text
ResourceScopeSelector
  - datasourceId
  - snapshotId
  - tableName
  - columnName
  - mode: datasource / snapshot / table / column
  - onlyPublishedSnapshot: true/false
  - includeAllOption: true/false
```

这个组件需要和 `AdminContextBar` 打通：顶部上下文选了数据源后，页面内默认继承；页面内切换数据源后，也更新全局上下文。这样后台使用时会形成“我正在处理哪个数据源”的稳定心智。

访问策略编辑器还应避免让用户直接理解底层字段：

- “表级策略”和“字段级策略”用分段控件切换。
- 表级策略只显示表选择，不显示字段输入框。
- 字段级策略必须先选表，再选字段。
- 脱敏策略只在字段级策略中出现。
- 行级过滤应提供“字段 + 操作符 + 值”的结构化编辑器，减少 SQL 片段手写。

#### 5.7 布局和排版应建立后台页面模板

目前页面布局存在一些不一致：有些页面使用顶部操作区 + 工具栏 + 表格，有些页面直接卡片堆叠；有些筛选项宽度固定但不统一；有些表格信息密度过高；部分页面把“主对象选择”和“列表筛选”混在同一行。

建议定义统一后台页面模板：

```text
页面标题区
  - 标题、说明、主操作按钮

上下文区
  - 数据源 / 快照 / 表 / 字段等主对象选择
  - 当前对象状态摘要

内容操作区
  - 搜索、筛选、批量操作

主内容区
  - 表格、详情、图谱、编辑器

辅助详情区
  - 抽屉或右侧面板，展示当前行详情、影响分析、操作历史
```

布局规则：

- 主对象选择不要和普通筛选混在一起。数据源、快照、表属于“上下文”，状态、严重级别、关键词属于“筛选”。
- 列表页默认采用“筛选条 + 表格 + 右侧详情抽屉”，不要把所有字段都挤在表格列里。
- 表格列优先级应固定：对象名称、状态、责任人/来源、关键指标、更新时间、操作。低频信息放详情抽屉。
- 批量操作只在选中行后出现，避免常驻按钮干扰。
- 高风险操作统一放到行操作菜单或二次确认里，按钮文案必须说明影响范围。
- 页面空态要告诉用户下一步做什么，例如“请先选择数据源”“当前快照暂无表数据”“该数据源尚未发布快照”。
- 表名、字段名、SQL、表达式用等宽字体；业务名称、状态、责任人用普通文本和标签。
- 表格不要滥用 `border`，治理后台更适合轻边界、高密度、清晰分组。

建议的三类页面布局：

| 页面类型 | 适用页面 | 推荐布局 |
| --- | --- | --- |
| 对象详情型 | 数据源详情、表详情、知识文档详情 | 左侧对象列表 + 右侧 Tabs 详情 |
| 工作队列型 | 治理问题、知识审核、访问申请、反馈审核 | 顶部筛选 + 中间队列 + 右侧详情/处理抽屉 |
| 配置矩阵型 | 权限中心、角色权限、行列级策略 | 左侧主体/数据源选择 + 中间权限矩阵 + 右侧最终决策说明 |

#### 5.8 可优先优化的具体前端点

建议按影响面排序：

1. **访问策略编辑器**：把表名/列名手输改成数据源、表、字段级联选择；行级过滤增加结构化编辑模式。
2. **血缘查看**：表和字段改成选择器，基于已发布快照或实体图加载选项。
3. **治理问题列表**：表名筛选改成表选择器；点表名/字段名可跳到表详情或字段详情。
4. **权限中心**：新增“最终权限预览”，选择用户和数据源后展示来自部门、角色、用户、临时授权的合并结果。
5. **后台首页**：从快捷入口改成数据源生命周期和待办队列。
6. **AdminContextBar**：升级为全局上下文选择器，让数据源、快照、知识文档选择被所有相关页面继承。

### 6. 落地建议：先改“层级和角色”，再改具体页面

不建议马上大规模重写所有 Vue 页面。更稳的顺序是：

1. **先定义角色矩阵**：明确每个岗位能看到哪些一级域、能执行哪些操作、默认拥有哪些数据范围。
2. **重组侧边栏**：把当前 11 个一级菜单收束为 6 个一级域，保留现有页面作为二级/三级入口。
3. **改造首页工作台**：从快捷入口改成待办、状态、风险和发布进度。
4. **选择一个核心对象做详情页试点**：优先做“数据源详情页”，把采集、表字段、快照、权限、质量、知识状态放到同一页。
5. **再做权限中心**：按“用户视角”和“数据源视角”展示最终权限决策，解决管理员不知道怎么分配权限的问题。
6. **最后清理重复页面**：当对象详情页稳定后，再移除或降级旧的表浏览器、快照列表、版本历史等独立入口。

### 8. 还需要补齐的设计文档内容

当前文档已经指出了后台的核心问题和重构方向，但如果要真正进入开发，还需要补齐以下内容，避免后续实施时再次变成“想到哪里改哪里”。

#### 8.1 岗位-权限矩阵明细

前文只定义了角色类型，还需要进一步落到权限矩阵。建议新增一张正式矩阵，覆盖：

- 一级导航可见性。
- 页面访问权限。
- 按钮/操作权限。
- 数据源默认访问范围。
- 审核/发布/回滚/审批权限。
- 是否允许查看 SQL、导出结果、配置脱敏、配置行级过滤。

示例：

| 角色 | 后台入口 | 数据源管理 | 治理处理 | 知识审核 | 权限审批 | 系统配置 | 查询数据范围 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 业务查询用户 | 否 | 否 | 否 | 否 | 提交申请 | 否 | 已授权数据源 |
| 数据分析师 | 是 | 只读 | 处理语义问题 | 提交/审核知识 | 否 | 否 | 授权业务域 |
| 数据管理员 | 是 | 管理 | 处理元数据问题 | 提交知识 | 否 | 否 | 负责数据源 |
| 数据安全管理员 | 是 | 只读 | 查看 | 否 | 审批 | 否 | 全部或指定域 |
| 治理负责人 | 是 | 查看 | 审核/发布 | 审核/发布 | 查看 | 否 | 负责域 |
| 系统运维管理员 | 是 | 技术配置 | 否 | 否 | 否 | 管理 | 不默认授予业务数据 |
| 超级管理员 | 是 | 管理 | 管理 | 管理 | 管理 | 管理 | 兜底权限 |

#### 8.2 数据源生命周期状态机定义

可询问状态需要从产品想法落到后端字段和计算规则。建议单独补一节状态机定义：

| 字段 | 建议来源 | 说明 |
| --- | --- | --- |
| `connectionStatus` | 数据源健康检查 | 连接是否可用 |
| `metadataStatus` | 元数据采集任务/快照 | 是否完成采集，是否有已发布快照 |
| `governanceStatus` | 治理问题/元数据状态 | 是否存在阻塞级问题 |
| `knowledgeStatus` | knowledge_doc / vector task | skills.md 是否发布、向量是否可召回 |
| `permissionStatus` | datasource_access / policy | 是否存在有效查询授权 |
| `askableStatus` | 聚合计算字段 | 是否可被自然语言查询 |
| `blockReasons` | 聚合计算字段 | 不可询问原因列表 |

建议新增一个聚合接口，供前台查询页和后台数据源详情页共用：

```text
GET /api/admin/datasources/{id}/readiness
GET /api/datasources/{id}/readiness
```

返回示例：

```json
{
  "datasourceId": 1,
  "askable": false,
  "stage": "KNOWLEDGE_INDEXING",
  "stageLabel": "知识库发布中",
  "progress": 75,
  "blockReasons": [
    {
      "code": "KNOWLEDGE_NOT_PUBLISHED",
```

#### 8.3 页面合并与路由迁移方案

当前文档提出了一级导航重组，但还需要列出旧页面如何迁移，避免一次性删除造成开发风险。

建议采用“三阶段迁移”：

1. **第一阶段：只重组菜单**  
   保留旧路由，先把入口归类到新的一级域下。

2. **第二阶段：建设对象详情页**  
   新增数据源详情页，把表字段、采集、快照、治理、知识、权限状态聚合进去。

3. **第三阶段：降级旧页面**  
   旧页面不立即删除，先变成详情页 Tab、二级入口或高级入口，确认稳定后再清理。

建议补一张迁移表：

| 旧入口 | 新归属 | 处理方式 |
| --- | --- | --- |
| 数据源管理 | 数据资产 | 保留列表，新增详情页 |
| 同步任务 | 数据资产 / 数据源详情 | 迁入数据源详情“采集任务”Tab |
| 表浏览器 | 数据资产 / 数据源详情 | 迁入“表字段”Tab |
| 快照列表 | 数据资产 / 数据源详情 | 迁入“快照版本”Tab |
| 质量看板 | 治理工作台 | 保留为概览 |
| 问题清单 | 治理工作台 | 改为待处理队列 |
| 字段标签/可信度/反馈审核 | 治理工作台或语义资产 | 按字段治理闭环合并 |
| 访问控制/策略编辑器 | 权限与合规 | 合并为权限中心 |
| 慢查询/操作日志/健康/AI 配置 | 系统运维 | 归入运维域 |

#### 8.4 统一组件清单

为了避免页面继续各写各的，需要把通用组件提前定义出来：

| 组件 | 用途 |
| --- | --- |
| `ResourceScopeSelector` | 数据源、快照、表、字段级联选择 |
| `DatasourceReadinessStepper` | 数据源生命周期和可询问进度 |
| `GovernanceTaskDrawer` | 治理问题、反馈、审核任务详情与处理 |
| `PermissionDecisionPanel` | 展示某用户/角色/部门对某数据源的最终权限 |
| `PolicyConditionBuilder` | 行级过滤结构化编辑器 |
| `AdminPageHeader` | 统一页面标题、说明、主操作 |
| `AdminFilterBar` | 统一筛选区布局 |
| `EmptyActionState` | 带下一步动作的空态 |

#### 8.5 验收标准

产品级整改也需要验收标准，不能只看“页面变好看”。建议补充：

- 普通业务用户只能看到可询问的数据源；不可询问时能看到明确原因和申请/联系入口。
- 管理员进入任一数据源详情页，能在 10 秒内判断该数据源当前卡在哪一步。
- 后台一级导航不超过 6 个业务域。
- 访问策略配置中不再需要手输表名和字段名。
- 血缘查看、治理问题筛选、访问策略编辑器都复用统一资源选择器。
- 权限中心能解释最终权限来源：部门、角色、用户、临时授权、显式拒绝。
- 至少一个核心流程可从工作台待办进入并闭环完成，例如“处理治理问题 -> 发布快照 -> 生成知识 -> 配置权限 -> 可询问”。

#### 8.6 风险和边界

这次优化要避免两个风险：

1. **不要只改菜单名称**  
   如果只是把旧页面换个分组，但没有数据源生命周期、待办和对象详情，割裂感仍然会存在。

2. **不要一次性重写所有页面**  
   应先做可询问状态、统一选择器、数据源详情页和权限中心四个高价值点。旧页面可以逐步降级，不必立即删除。

推荐第一轮开发范围：

```text
1. 数据源 readiness 聚合接口
2. 数据源生命周期进度组件
3. ResourceScopeSelector 统一选择器
4. 访问策略编辑器改造
5. 血缘查看改造
6. 后台首页增加数据源可询问状态和待办
```

阶段设计原则：不建议在 `datasource` 表中新增一个静态“当前阶段”字段。数据源是否可询问由连接状态、发布快照、治理阻塞问题、skills.md 发布/向量化和权限配置共同决定，这些事实分散在不同业务表中。若把阶段直接落到 `datasource`，很容易出现“字段显示已发布，但知识或权限已经变化”的状态漂移。当前阶段应作为 readiness 聚合结果实时计算；只有将来需要流程审批留痕时，再新增独立的流程实例/流程事件表记录历史，而不是覆盖数据源主表。

- 数据源管理页新增“上线阶段”列：
  - 列表直接展示当前聚合阶段和进度。
  - 每行提供“流程”入口，进入单个数据源的上线流程详情。
- 新增数据源上线流程详情页：

  - 每一步展示责任角色、完成状态、说明和下一步入口。
